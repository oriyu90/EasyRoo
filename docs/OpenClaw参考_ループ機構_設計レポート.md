# OpenClaw のループ機構と EasyRoo への適用 — 設計レポート

- 対象: EasyRoo v1.0
- 調査対象: [OpenClaw](https://github.com/openclaw/openclaw) / [公式ドキュメント](https://docs.openclaw.ai/)
- 作成日: 2026年7月22日
- 著者: yuki_orita（実装: Claude）

---

## 0. 要約

OpenClaw は「多数のメッセージング経路から届く会話を、複数のエージェントに振り分けて
長期間動かし続ける」ためのゲートウェイである。そのループ機構が評価されているのは、
**LLM の有限なコンテキスト長という制約の上で、会話の一貫性をどう保つか**という
一点に対して、実運用で必要な対策が層状に積まれているためである。

EasyRoo は会話アプリではなく**スケジュール実行のルーティーンランナー**であるため、
OpenClaw の機構をそのまま持ち込むことはできない。本レポートでは 12 の機構を分解し、
**EasyRoo の問題設定に照らして 5 件を採用、7 件を不採用**と判断した。
不採用のものについても、なぜ不要かを明記する。

採用する 5 件は、いずれも **EasyRoo に既に存在する弱点を塞ぐもの**である。

## 1. OpenClaw のループ機構の分解

### 1.1 ターンのライフサイクル

> "intake, context assembly, model inference, tool execution, streaming, persistence"

`agent` RPC はパラメータ検証・セッション解決・メタデータ永続化を行い、
**即座に `{ runId, acceptedAt }` を返す**。実行の完了を待たない。
待機は別 RPC (`agent.wait`) に分離されている。

### 1.2 実行レーンによる直列化

> "Runs are serialized per session key (session lane) and optionally through a global lane,
> preventing tool/session races."

セッション単位のレーンと、任意でグローバルレーンにより実行を直列化する。
同一セッションに対する同時実行が起きないことを構造で保証している。

さらにトランスクリプト書き込みには**セッション書き込みロック**があり、
最大 60 秒待ってから「busy」と判定する。

### 1.3 実行中に届いたメッセージの扱い（キューモード）

> "Messaging channels select queue modes (steer/followup/collect/interrupt) that feed this lane system."

エージェントが動いている最中に新しいメッセージが来たとき、どう振る舞うかを選べる。

| モード | 挙動 |
| --- | --- |
| `steer` | 実行中のターンに割り込んで方向を変える |
| `followup` | 現在のターンの後に続けて処理する |
| `collect` | まとめて溜め、次のターンで一括処理 |
| `interrupt` | 実行中のターンを中断して新しい要求を処理 |

**これが OpenClaw のループで最も特徴的な部分である。**
「忙しいから拒否」ではなく、重複をポリシーとして扱う。

### 1.4 多層のタイムアウト

| 層 | 既定値 | 役割 |
| --- | --- | --- |
| `agent.wait` | 30秒 | 待つのをやめるだけ。実行は止めない |
| エージェント実行時間 | 172800秒 (48時間) | `runEmbeddedAgent` の中断タイマー |
| モデル無応答 | 120秒 (Cloud) / 300秒 (自己ホスト) | チャンクが来なくなったら中断 |
| プロバイダHTTP | — | 接続・ヘッダ・本文・ストリーム停滞を個別に監視 |

「待機のタイムアウト」と「実行のタイムアウト」を**別物として分けている**点が重要。

### 1.5 自動圧縮（オートコンパクション）

2 つの契機で発動する。

1. **オーバーフロー復帰** — モデルがコンテキスト超過エラーを返したら、圧縮して**再試行**する
2. **閾値維持** — ターン成功後、現在のコンテキストが
   > "the model window minus OpenClaw's built-in headroom for prompts and the next model output"
   を超えていたら圧縮する

加えて 2 つの防御がある。

- **事前ローカル圧縮**: `maxActiveTranscriptBytes`（例 `20mb`）による物理サイズ基準
- **ターン中プリチェック**: ツール結果を追記した結果コンテキストが収まらなくなったら、
  プロンプト送信を**中止**する

既定設定:

```json5
compaction: { enabled: true, keepRecentTokens: 20000 }
```

### 1.6 圧縮前のメモリフラッシュ ★

圧縮の直前に、**サイレントなエージェントターン**を 1 回挟み、
重要な情報をワークスペースへ書き出させる。

| 設定 | 既定 | 役割 |
| --- | --- | --- |
| `enabled` | `true` | 有効化 |
| `softThresholdTokens` | `4000` | 圧縮閾値の何トークン手前で発火するか |
| `forceFlushTranscriptBytes` | 未設定 | このサイズに達したら強制 |

出力が `NO_REPLY` で始まる場合は配信を抑止する。1 圧縮サイクルにつき 1 回だけ動く。
ワークスペースが読み取り専用なら丸ごとスキップする。

**この機構の意義**: 圧縮は必ず情報を失う。失う前に「何を残すべきか」を
エージェント自身に判断させて永続化する。要約に頼るのではなく、
**構造化された記憶へ退避させる**という発想である。

### 1.7 圧縮時の分割境界の保護

> "If the token-share split would land between a tool call and its result,
> OpenClaw shifts the boundary to the assistant tool-call message instead of separating the pair."

ツール呼び出しとその結果を分断しない。分断すると OpenAI 互換 API が 400 を返すため。

### 1.8 サブエージェント（`sessions_spawn`）★

| パラメータ | 意味 |
| --- | --- |
| `context` | `"isolated"`（既定）= まっさらから開始 / `"fork"` = 親のトランスクリプトを共有 |
| `model` / `thinking` | 子側の上書き |
| `sandbox` | サンドボックス強制 |

`{ runId, childSessionKey }` を**即座に返す**。
子は最初のターンで `[Subagent Task]` として委任内容を受け取る。

**深さ制限**: 既定の末端サブエージェントはセッションツールを持たない。
`sessions_spawn` を持つのは深さ 1 のオーケストレータのみ。無限の入れ子を防いでいる。

**親フォークの上限**: 親のブランチが内部上限（現在 10 万トークン）を超える場合、
継承させずに**子のコンテキストを隔離**する。

### 1.9 セッションのリセット方針

`manual`（`/new`）/ `daily`（既定 4 時境界）/ `idle`（無活動 N 分）の 3 方式。
ハートビートや cron などのシステムイベントはセッション行を更新するが、
**鮮度タイマーは延長しない**。周辺の自動活動が古いセッションを生き永らえさせないため。

### 1.10 サイレントターン（`NO_REPLY`）

アシスタント出力が `NO_REPLY` で始まる場合、返信を抑止する。
背景処理のための仕組み。`2026.1.10` からは部分チャンクの段階でも抑止する。

### 1.11 ストア保守とディスク予算

| 設定 | 既定 |
| --- | --- |
| `pruneAfter` | `30d` |
| `maxEntries` | `500` |
| `maxDiskBytes` | `10gb` |
| `highWaterBytes` | max の 80% |

超過時は、チェックポイント領域の回収 → 古いアーカイブ削除 →
古いセッションから圧縮アーカイブ化して削除、の順で回収する。

### 1.12 サンドボックスの階層

> "Default: tools run on the host for the `main` session"
> "set `agents.defaults.sandbox.mode: \"non-main\"` to run non-`main` sessions inside sandboxes"

`main`（=自分だけ）はホスト直、それ以外は Docker / SSH / OpenShell のサンドボックス。
許可ツールと拒否ツールをサンドボックス設定で指定する。

## 2. EasyRoo との構造差

適用可能性を判断する前に、両者の問題設定の違いを明確にする。

| 観点 | OpenClaw | EasyRoo |
| --- | --- | --- |
| 起動源 | 人間からのメッセージ（20以上の経路） | **スケジュール**または手動実行 |
| 実行単位 | 会話セッション（無期限に継続） | ルーティーン実行（1回で完結） |
| 文脈の寿命 | セッションが続く限り無限に伸びる | 1 実行で閉じる。実行間は STATE/JOURNAL で繋ぐ |
| 実行中の外部入力 | 頻繁（人が話しかける） | **無い**（ただしスケジュールの再発火はある） |
| 出力先 | チャット経路へ返信 | ログと実測値。返信先は無い |
| 信頼境界 | 送信者が他人でありうる → ペアリング/サンドボックス | **利用者本人のみ** |
| 並行性 | セッションが多数並行 | ルーティーンが最大 3 並行 |

**最大の差は「実行中に人が話しかけてこない」こと。**
これにより steer / followup / NO_REPLY / sessions_send といった
「会話の交通整理」に属する機構は、EasyRoo では役割を持たない。

**逆に共通する最大の課題は「コンテキスト長との戦い」である。**
ここに属する機構は、そのまま EasyRoo の弱点に対応する。

## 4. 採用する 5 機構の設計

### A. トリガ重複ポリシー（1.2 / 1.3 由来）

**現状の問題**
毎時実行のルーティーンが 70 分かかった場合、次の発火は
`「〇〇」は既に実行中です` で**捨てられる**。利用者には通知も残らない。
「1 時間ごと」と設定したのに実質 2 時間ごとになり、しかも気づけない。

**設計**
ルーティーンごとに `overlapPolicy` を持たせる。

| 値 | 挙動 | 用途 |
| --- | --- | --- |
| `skip`（既定） | 実行中なら今回は見送る。**ただし記録は残す** | 冪等でない処理 |
| `queue` | 現在の実行の完了後に 1 回だけ実行する | 取りこぼしたくない処理 |
| `restart` | 実行中のものを停止して新しく始める | 最新状態だけが意味を持つ処理 |

OpenClaw の `collect` に相当するのが `queue`、`interrupt` が `restart`。
`steer`（実行中のターンへの割り込み）は、外部入力が無い EasyRoo では意味を持たないため
採用しない。

キューは**ルーティーンあたり 1 件まで**とする。積み上げると、遅延の原因が解消した後に
一斉に走り出して事態を悪化させるため（OpenClaw のレーンも同様に直列化で抑えている）。

### B. 圧縮前メモリフラッシュ（1.6 由来）★ 最重要

**現状の問題**
EasyRoo は既に「捨てずに要約して畳む」圧縮を持つ。しかし要約は**平坦な文章**であり、
次の圧縮でさらに要約され、劣化していく。実行が長引くほど序盤の事実は薄れる。

一方 EasyRoo には **STATE.md という構造化された永続領域が既にある**。
圧縮で失う前にそこへ退避させれば、要約の劣化と無関係に情報が残る。
**OpenClaw の機構と EasyRoo の既存資産が正確に噛み合う。**

**設計**
圧縮閾値の手前（`softThresholdRatio`、既定 0.9）に達したら、
圧縮の**前に**サイレントターンを 1 回挟む。

1. 現在の履歴と現在の STATE をモデルへ渡す
2. 「この先の作業に必要な事実だけを STATE として書き直せ」と指示する
3. 返ってきた内容で STATE を更新する（`state_write` と同じ経路）
4. その後、通常どおり圧縮する

制約（OpenClaw に倣う）:
- **1 圧縮サイクルにつき 1 回**だけ（`memoryFlushedForCompaction` フラグ）
- **記憶が無効なルーティーンではスキップ**（OpenClaw の read-only ワークスペースと同じ扱い）
- 失敗しても実行は継続する（フラッシュは最適化であって必須処理ではない）

### C. コンテキスト超過からの復帰（1.5a 由来）

**現状の問題**
モデルが `context_length_exceeded` を返すと、`llm.chat` が例外を投げ、
実行がそのまま失敗する。ここまでの作業がすべて無駄になる。

**設計**
LLM 応答のエラーがコンテキスト超過を示す場合（HTTP 400 かつ本文に
`context length` / `too many tokens` / `maximum context` 等を含む）、
**強制的に圧縮して 1 回だけ再試行**する。再試行してなお失敗したら通常どおり失敗させる。

閾値ベースの圧縮（D）が正しく効いていれば滅多に発動しないが、
**モデル窓の推定が外れたときの最後の受け皿**として意味を持つ。

### D. トークン基準の予算管理（1.5b 由来）

**現状の問題**
履歴上限が**固定 60,000 文字**。128k 窓のモデルでは過剰に切り詰め、
8k 窓のモデルでは足りずに溢れる。日本語と英語で 1 文字あたりのトークン数も違う。

**設計**
- プロバイダごとに `contextTokens`（モデルの窓、既定 32768）を設定可能にする
- 予算 = `contextTokens − 出力用の余白 − プロンプト用の余白`
  （OpenClaw の "model window minus built-in headroom" と同じ考え方）
- トークン数は、**実測値があればそれで較正**する。
  EasyRoo は既に `usage.prompt_tokens` を記録しているため、
  「直近の実測トークン ÷ その時の履歴文字数」で 1 文字あたりのトークン数を推定できる。
  実測が無い初回は、日本語混在を考慮した保守的な既定値を使う。

較正を入れる理由: トークナイザはモデルごとに違い、EasyRoo は任意の OpenAI 互換
エンドポイントを相手にするため、正確なトークナイザを内蔵できない。
**実測から逆算するのが、依存を増やさずに精度を上げる唯一の方法である。**

### E. サブラン（隔離された文脈での部分実行）（1.8 由来）

**現状の問題**
「20 個のファイルを 1 つずつ処理する」ような手順書では、
1 ファイル分の作業ログがすべて同じ文脈に積まれる。20 個目に着く頃には
序盤が圧縮で失われ、かつトークンを浪費している。

**設計**
`spawn_subrun` ツールを追加する。

| パラメータ | 内容 |
| --- | --- |
| `task` | 委任する作業内容（文字列） |
| `context` | `isolated`（既定）のみ。`fork` は採用しない |

- 子は**まっさらな文脈**で開始し、親の手順書・注意書き・禁止規則・作業ディレクトリを継承する
- 子のステップ予算は親の残りステップ数の範囲内（既定 `min(10, 親の残り)`）
- 子は `finish` で終了し、**その要約だけが親へ返る**
- **入れ子は 1 段まで**（OpenClaw の「深さ 1 のみが `sessions_spawn` を持つ」に倣う）
- 子は記憶ツール（`state_write` / `journal_append`）を持たない。
  記憶の一貫性は親が責任を持つ

`context: "fork"` を採用しない理由: 親の文脈を継承すると、
サブランの目的である「文脈の隔離」が失われる。OpenClaw が
「親ブランチが 10 万トークンを超えたら隔離に切り替える」としているのも同じ理由であり、
EasyRoo の規模では常に隔離で構わない。

## 5. 不採用とした機構とその理由

| 機構 | 不採用の理由 |
| --- | --- |
| `steer` / `followup` キューモード | 実行中に人が話しかけてくる経路が無い。EasyRoo の重複は「スケジュール再発火」だけであり、`skip`/`queue`/`restart` で尽きる |
| `NO_REPLY` サイレントターン | 返信の配信先が無いため、抑止すべき出力が存在しない。B のフラッシュは元々ログにしか出ない |
| セッションリセット（daily / idle） | EasyRoo の実行は 1 回で閉じる。「古いセッションが居座る」問題が起きない |
| ディスク予算・アーカイブ | OpenClaw は 10GB 予算を前提とする規模。EasyRoo は実行ログ 200 件・JOURNAL 60 件で数 MB に収まる。**今の規模に対して過剰** |
| ターン中プリチェック | EasyRoo はループ先頭で毎回圧縮判定する。ツール結果追記の直後が次のループ先頭なので実質等価 |
| モデル無応答タイムアウト | EasyRoo は非ストリーミングのため、リクエストタイムアウトが同じ役割を果たす |
| サンドボックス階層（Docker/SSH） | **意図的に見送る。** OpenClaw がサンドボックスを必要とするのは、他人からのメッセージでエージェントが動きうるためである。EasyRoo の実行を起動できるのは利用者本人のみで、信頼境界が違う。Docker 依存を持ち込むことは「ランタイム依存ゼロ」という設計方針（設計書 1.2）とも正面から衝突する。現在の**禁止規則 8 カテゴリ + 作業ディレクトリ + タイムアウト + 注意書き**という多層防御を、この用途における妥当な水準と判断する |

| `src/shared/i18n.js` | 追加文言（日英） |
| `src/renderer/app.js` | 重複ポリシーの UI、実測値への圧縮回数表示 |
| `bin/easyroo.js` | `--overlap` オプション |
| `test/run-tests.js` | 各機構の検証 |

### 6.2 実装順序

1. **D. 予算管理** — 他の機構の土台になるため最初に置く
2. **C. 超過復帰** — D の推定が外れた場合の受け皿。D と対で意味を持つ
3. **B. メモリフラッシュ** — 圧縮経路が固まってから差し込む
4. **A. 重複ポリシー** — 実行制御層。他と独立して進められる
5. **E. サブラン** — 最も影響範囲が広いため最後

### 6.3 検証方針

| 機構 | 検証方法 |
| --- | --- |
| A | 実行中に再発火させ、3 方針それぞれの挙動を確認。キューが 1 件で頭打ちになること |
| B | 閾値手前で STATE が更新されること。1 サイクル 1 回であること。記憶無効時にスキップされること |
| C | コンテキスト超過エラーを模擬し、圧縮して再試行されること。2 回目の失敗で通常失敗すること |
| D | 実測トークンによる較正が働くこと。窓の異なるモデルで予算が変わること |
| E | 子が隔離文脈で動き、要約だけが親に返ること。入れ子が 1 段で止まること |
| 全体 | ESLint エラー 0、既存 137 テストの非回帰、実 LLM でのエンドツーエンド実行 |

### 6.4 想定リスク

| リスク | 対策 |
| --- | --- |
| メモリフラッシュが LLM 呼び出しを増やしコストが上がる | 1 圧縮サイクル 1 回に制限。長い実行でしか発動しない |
| トークン推定の誤差 | 較正 + C の超過復帰を受け皿にする。推定は保守側（多め）に倒す |
| サブランが親のステップ予算を食い潰す | 子の予算を親の残りから引く。入れ子 1 段制限 |
| `restart` ポリシーで作業が中途半端に終わる | 停止は既存の中断機構を使い、記録に残す。既定は `skip` のまま |
